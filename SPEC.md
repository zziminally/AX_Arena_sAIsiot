## 1. 프로젝트 개요 및 목표

### 배경

올림플래닛은 체험형 광고 마케팅(XR, 이머시브 콘텐츠)을 전문으로 하는 애드테크 기업으로, 1,000건 이상의 캠페인을 수행하며 방대한 제안서 자산을 보유하고 있다. 그러나 **신규 제안서 작성 시 이 자산을 빠르게 참조·재구성하는 워크플로우가 부재**하여, 담당자가 매번 수동으로 과거 자료를 찾고 새로 조합하는 데 과도한 시간을 소비한다.

### 핵심 문제 정의

```
[현재 상태]
신규 제안 요청 → 담당자가 수동으로 과거 제안서 탐색 (수십~수백 개 중) → 적합한 레퍼런스 선별 →
구조·톤·섹션 수동 재구성 → 초안 완성 (수 시간 ~ 수일 소요)

[목표 상태]
신규 제안 요청 → 시스템이 자동으로 최적 레퍼런스 추천 →
레퍼런스 기반 제안서 초안 자동 구성 (PPT 형식) → 담당자 검토·보완 (수십 분 내)
```

### 서비스명

**ProposalPilot** — "검색을 넘어서, 참조와 재구성까지"

### 목표

| 구분 | 목표 |
|------|------|
| 기능 목표 | 임의의 신규 제안 요청(시나리오)에 대해 최적 레퍼런스 top-3 추천 + PPT 초안 자동 구성 |
| 품질 목표 | 추천 제안서의 산업군·프로젝트 유형 매칭 정확도 ≥ 80% (검증 시나리오 3종 기준) |
| 효율 목표 | 초안 구성 소요 시간 기존 대비 70% 이상 단축 |
| 확장 목표 | 신규 제안서 축적 시 인덱스에 자동 편입 가능한 구조 |

```python

    "고객사 유형":       "client_type",       # "자동차 제조 기업" | "프리미엄 가전 브랜드" | ...
    "제안 목적":        "proposal_purpose",  # "신차 런칭 통합 체험 전략 수립" | ...
    "프로젝트 유형":     "project_type",      # "브랜드/캠페인 전략" | "프로젝트/공간 기획" |

    "제안 단계":        "proposal_stage",    # "전략 제안" | "공간 기획 제안" | "콘텐츠 설계 제안" |

    "주요 니즈 키워드":  "key_needs_keywords",# "신차 런칭|통합 체험 캠페인|XR 시승" → 리스트 파싱
```

### 3.4 청킹(Chunking) 전략

섹션 단위 청킹 채택 — 제안서의 명확한 섹션 구조를 그대로 활용하므로 의미 단위 훼손 없음.

```python
# 청크 단위: 섹션 구분 슬라이드 이후 연속된 콘텐츠 슬라이드 텍스트 합산
{
    "chunk_id":      "OP-S1-01_sec_04",           # doc_id + 섹션 인덱스
    "doc_id":        "OP-S1-01",
    "section_name":  "추진 전략 또는 제안 프레임",
    "section_order": 4,
    "text":          "...",                        # 해당 섹션 모든 슬라이드 텍스트 병합
    "metadata": {                                  # CSV 메타데이터 전체 포함
        "industry":          "자동차",
        "project_type":      "브랜드/캠페인 전략",
        "proposal_purpose":  "신차 런칭 통합 체험 전략 수립",
        "tone_and_manner":   "프리미엄·전략적·설득형",
        "emphasized_values": ["브랜드 신뢰", "기술 이해", "리드 전환"],
        ...
    }
}
```

**섹션 구분 감지 로직**: 슬라이드 텍스트에서 `OLIMPLANET`, `© OLIMPLANET Reference Asset`, 페이지 번호를 제거한 뒤 짧은 단일 텍스트(≤ 30자)만 남으면 섹션 구분 슬라이드로 판단.

## 4. 추천(검색) 모듈 설계

### 4.1 전체 흐름

```
사용자 입력
   │
   ▼
[Step 1] Request Analyzer (LLM)
   └─ 자유 서술 → 구조화된 필터 + 검색 쿼리 추출
   └─ 출력: { industry, client_type, project_type, purpose, query_text }
   │
   ▼
[Step 2] Hard Filter (메타데이터 필터링)
   └─ industry 일치 OR 유사 산업군 (예: 자동차↔모빌리티)
   └─ project_type 필터 (XR, 팝업, 온라인캠페인 등)
   └─ doc_year ≥ 현재-3년 (기본값, 조정 가능)
   └─ 후보군 → N개 (전체 20개 중 5~10개)
   │
   ▼
[Step 3] Semantic Search (벡터 유사도)
   └─ query_text 임베딩 생성
   └─ 후보군 내 cosine similarity 계산
   └─ 상위 top-k=5 선발
   │
   ▼
[Step 4] Hybrid Scoring & Re-ranking
   └─ 최종 점수 = α × metadata_score + (1-α) × semantic_score
   └─ α = 0.4 (초기값, 튜닝 가능)
   └─ metadata_score: 세부 필드 매칭 수 기반 (최대 1.0)
   └─ semantic_score: cosine similarity (0~1)
   └─ top-3 최종 선발
   │
   ▼
[Step 5] 출력: top-3 레퍼런스 + 유사도 근거 로그
```

### 4.2 메타데이터 필터링 전략

**하드 필터 (Hard Filter)** — 벡터 검색 전 후보군 제한

| 필드 | 필터 유형 | 로직 |
|------|----------|------|
| `industry` | 소프트 매칭 | 정확 일치 우선, 없으면 유사 산업군 허용 |
| `project_type` | 소프트 매칭 | 요청 키워드 → 3.2절 매핑표 기준 후보 유형 선별 |
| `doc_year` | 범위 | 최근 3년 기본 (현재 데이터 전부 2025년이므로 사실상 전체) |

**산업군 유사성 맵** (실제 데이터 기준 확장)

```python
INDUSTRY_SIMILARITY_MAP = {
    "자동차":     ["모빌리티", "자동차 제조", "전기차"],
    "가전":       ["전자", "생활가전", "프리미엄 가전"],
    "식음료":     ["F&B", "식품", "음료", "외식", "프랜차이즈", "레스토랑"],
    "확장 레퍼런스": [],  # 산업군 불문 활용 가능한 프레임워크 문서
    "부동산":     ["분양", "시행사", "체험관"],
}
```

> **확장 레퍼런스(S4) 처리**: 산업군 매칭 실패 시 fallback 후보군으로 항상 포함. 온오프라인 통합 여정, XR 모듈 재사용, KPI 설계 등 범용 프레임워크 문서이므로 어떤 산업 요청에도 보조 참조 가능.

### 4.4 유사도 계산 및 top-k 선정 로직

```python
def hybrid_score(doc_metadata, query_metadata, semantic_sim):
    """
    실제 CSV 필드를 기준으로 메타데이터 매칭 점수 계산.
    가중치가 다른 이유: industry 불일치는 제안서 적합성에 치명적,
    project_type은 비슷한 유형이 많으므로 부분 점수 허용.
    """
    score = 0.0

    # industry: 가중치 높음 (정확 일치 0.5, 유사 산업군 0.25)
    if doc_metadata["industry"] == query_metadata.get("industry"):
        score += 0.5
    elif doc_metadata["industry"] in INDUSTRY_SIMILARITY_MAP.get(
            query_metadata.get("industry", ""), []):
        score += 0.25
    elif doc_metadata["industry"] == "확장 레퍼런스":
        score += 0.1  # fallback 가산점

    # project_type: 키워드 겹침 기반 부분 점수
    q_type_keywords = PROJECT_TYPE_KEYWORDS.get(query_metadata.get("project_type", ""), set())
    d_type_keywords = PROJECT_TYPE_KEYWORDS.get(doc_metadata.get("project_type", ""), set())
    overlap = len(q_type_keywords & d_type_keywords) / max(len(q_type_keywords), 1)
    score += 0.3 * overlap

    # key_needs_keywords: 교집합 비율
    q_kw = set(query_metadata.get("key_needs_keywords", []))
    d_kw = set(doc_metadata.get("key_needs_keywords", []))
    if q_kw:
        score += 0.2 * len(q_kw & d_kw) / len(q_kw)

    metadata_score = min(score, 1.0)

    alpha = 0.4  # 메타데이터 40%, 시맨틱 60%
    return alpha * metadata_score + (1 - alpha) * semantic_sim

# top-k 선정: 하이브리드 점수 기준 상위 3개
# 동일 doc_id의 복수 청크는 최고 점수 청크만 대표로 선택 (max pooling)
# 단, 같은 project_type만 top-3를 차지하지 않도록 다양성 보장
#   → 동일 project_type 최대 2개 제한
```

## 8. 평가 기준 대응표

### 기업부문 심사 (70%)

| 평가 항목 | 배점 | 이 설계의 대응 |
|----------|------|-------------|
| 현업 문제 이해도 | 15점 | 올림플래닛의 실제 업무 플로우(제안→수주→구현)를 분석하여, "자료 부재"가 아닌 "자산 재연결 부재" 문제를 정확히 진단. 담당자 시간 낭비 구체화 (수 시간 → 수십 분 목표). |
| 실제 도입 가능성 | 30점 | 기존 제안서 파일(PDF/PPTX/DOCX) 그대로 인입 가능. 단일 명령어 인덱싱. 결과물이 기존 작업 도구(PowerPoint)와 동일한 포맷. 추가 인프라 불필요 (로컬 실행). |
| 문제 해결의 참신성 | 40점 | 단순 키워드 검색이 아닌 메타데이터 필터링 + 시맨틱 검색의 하이브리드 구조. 레퍼런스 '참조'에 그치지 않고 신규 요청 맥락에 맞게 '재구성'하는 2단계 구조. 결과물이 바로 사용 가능한 PPT. |
| 업무 효율·비용 절감 | 15점 | 초안 작성 소요 시간 70% 단축 목표. 레퍼런스 탐색 자동화로 담당자 집중력을 아이디어·보완에 투입 가능. 반복 작업 구조화로 팀 전체 노하우 공유 효과. |

### AI전문가 심사 (30%)

| 평가 항목 | 배점 | 이 설계의 대응 |
|----------|------|-------------|
| AI 활용 적합성 | 20점 | LLM 2단계 호출(구조화 파싱 + 섹션별 생성)로 역할 분리. 임베딩 기반 시맨틱 검색으로 키워드 검색 한계 극복. 제안서 특성(구조화된 장문 한국어)에 적합한 Claude claude-sonnet-4-6 선택. |
| 기술 구현 완성도 | 20점 | 검증 시나리오 3종에 대해 end-to-end 실행 가능한 데모 제공. 오류 처리, 로깅, retrieval_log.json 근거 추적 포함. |
| 데이터·프롬프트·워크플로우 설계 | 40점 | 메타데이터 필드 11개 정의, 하이브리드 스코어링 수식 명시, 청킹 전략 근거 설명. Prompt A/B 구조 설계, 역할·규칙·출력 형식 명시. 5단계 온라인 파이프라인 단계별 입출력 정의. |
| 기술 확장 가능성 | 20점 | 신규 제안서 추가 시 `ingest.py` 재실행으로 자동 편입. 벡터 스토어를 Pinecone으로 교체 가능(인터페이스 분리). 임베딩/LLM 모델 교체 시 설정값만 변경. 웹 API 전환 시 FastAPI 래핑 구조 준비. |

| 리스크 | 가능성 | 영향 | 대안 |
|--------|--------|------|------|
