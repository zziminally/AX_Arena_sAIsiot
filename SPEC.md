## 1. 프로젝트 개요 및 목표

### 배경

올림플래닛은 체험형 광고 마케팅(XR, 이머시브 콘텐츠)을 전문으로 하는 애드테크 기업으로, 1,000건 이상의 캠페인을 수행하며 방대한 제안서 자산을 보유하고 있다. 그러나 **신규 제안서 작성 시 이 자산을 빠르게 참조·재구성하는 워크플로우가 부재**하여, 담당자가 매번 수동으로 과거 자료를 찾고 새로 조합하는 데 과도한 시간을 소비한다.

### 핵심 문제 정의

```
[현재 상태]
신규 제안 요청 → 담당자가 수동으로 과거 제안서 탐색 (수십~수백 개 중) → 적합한 레퍼런스 선별 →
구조·톤·섹션 수동 재구성 → 초안 완성 (수 시간 ~ 수일 소요)

[목표 상태]
신규 제안 요청 → 시스템이 자동으로 최적 레퍼런스 추천 →
레퍼런스 기반 제안서 초안 자동 구성 (PPT 형식) → 담당자 검토·보완 (수십 분 내)
```

### 서비스명

**ProposalPilot** — "검색을 넘어서, 참조와 재구성까지"

### 목표

| 구분 | 목표 |
|------|------|
| 기능 목표 | 임의의 신규 제안 요청(시나리오)에 대해 최적 레퍼런스 top-3 추천 + PPT 초안 자동 구성 |
| 품질 목표 | 추천 제안서의 산업군·프로젝트 유형 매칭 정확도 ≥ 80% (검증 시나리오 3종 기준) |
| 효율 목표 | 초안 구성 소요 시간 기존 대비 70% 이상 단축 |
| 확장 목표 | 신규 제안서 축적 시 인덱스에 자동 편입 가능한 구조 |

### 모듈 책임 요약

| 모듈 | 역할 | 입력 | 출력 |
|------|------|------|------|
| Ingestor | 문서 파싱 및 정규화 | PDF/PPTX/DOCX + 메타데이터 JSON | 정규화된 텍스트 청크 + 메타데이터 |
| Embedding Model | 텍스트 → 벡터 | 텍스트 청크 | 부동소수점 벡터 (1536-dim) |
| Vector Store | 벡터·메타데이터 저장 및 검색 | 벡터 + 필터 쿼리 | 유사도 순위 문서 목록 |
| Request Analyzer | 자유 서술 → 구조화 쿼리 | 사용자 입력 | 구조화된 필터 + 쿼리 텍스트 |
| Hybrid Retriever | 메타데이터 필터 + 벡터 검색 | 구조화 쿼리 | top-k 레퍼런스 문서 |
| Draft Generator | LLM 기반 초안 생성 | top-k 문서 + 요청 | 섹션별 텍스트 초안 |
| PPT Assembler | 슬라이드 생성 | 초안 텍스트 + 템플릿 | .pptx 파일 |

## 3. 데이터 스키마 설계

### 3.1 메타데이터 필드 정의

```json
{
  "doc_id": "string",           // 고유 식별자 (예: OP-2024-001)
  "title": "string",            // 제안서 제목 (익명화 가능)
  "industry": "string",         // 산업군 (자동차, 가전, 식음료, 패션, 금융, 게임, 등)
  "client_type": "string",      // 고객사 유형 (대기업, 중견기업, 스타트업, 공공기관)
  "client_scale": "string",     // 고객사 규모 (국내대기업, 글로벌, 중소)
  "project_type": "string",     // 프로젝트 유형 (XR체험, 팝업스토어, 온라인캠페인, 오프라인이벤트, 하이브리드)
  "proposal_stage": "string",   // 제안 단계 (1차제안, 2차제안, 최종제안, 수주후기획)
  "included_sections": ["string"], // 포함 섹션 목록 (["배경분석","솔루션개요","캠페인기획","예산","레퍼런스"])
  "emphasized_values": ["string"], // 강조 가치 (["몰입감","기술혁신","브랜드경험","바이럴","데이터수집"])
  "proposal_purpose": "string", // 제안 목적 (신제품출시, 브랜드리뉴얼, 캠페인활성화, 신규고객유치)
  "doc_year": "integer",        // 문서 연도 (2020-2025)
  "key_needs_keywords": ["string"], // 주요 니즈 키워드 (["XR","체험마케팅","팝업","메타버스","AR필터"])
  "tone_and_manner": "string",  // 톤앤매너 (프리미엄/테크니컬/감성적/트렌디/신뢰감)
  "campaign_channel": ["string"], // 캠페인 채널 (["온라인","오프라인","소셜미디어","OOH"])
  "outcome_metrics": ["string"],   // 성과 지표 (["노출수","체험자수","전환율","브랜드인지도"])
  "file_path": "string",        // 원본 파일 경로
  "content_summary": "string",  // LLM이 생성한 100자 이내 요약
  "embedding_id": "string"      // Vector Store 내 레코드 ID
}
```

## 4. 추천(검색) 모듈 설계

### 4.1 전체 흐름

```
사용자 입력
   │
   ▼
[Step 1] Request Analyzer (LLM)
   └─ 자유 서술 → 구조화된 필터 + 검색 쿼리 추출
   └─ 출력: { industry, client_type, project_type, purpose, query_text }
   │
   ▼
[Step 2] Hard Filter (메타데이터 필터링)
   └─ industry 일치 OR 유사 산업군 (예: 자동차↔모빌리티)
   └─ project_type 필터 (XR, 팝업, 온라인캠페인 등)
   └─ doc_year ≥ 현재-3년 (기본값, 조정 가능)
   └─ 후보군 → N개 (전체 20개 중 5~10개)
   │
   ▼
[Step 3] Semantic Search (벡터 유사도)
   └─ query_text 임베딩 생성
   └─ 후보군 내 cosine similarity 계산
   └─ 상위 top-k=5 선발
   │
   ▼
[Step 4] Hybrid Scoring & Re-ranking
   └─ 최종 점수 = α × metadata_score + (1-α) × semantic_score
   └─ α = 0.4 (초기값, 튜닝 가능)
   └─ metadata_score: 세부 필드 매칭 수 기반 (최대 1.0)
   └─ semantic_score: cosine similarity (0~1)
   └─ top-3 최종 선발
   │
   ▼
[Step 5] 출력: top-3 레퍼런스 + 유사도 근거 로그
```

### 4.2 메타데이터 필터링 전략

**하드 필터 (Hard Filter)** — 벡터 검색 전 후보군 제한

| 필드 | 필터 유형 | 로직 |
|------|----------|------|
| `industry` | 소프트 매칭 | 정확 일치 우선, 없으면 유사 산업군 허용 |
| `project_type` | 소프트 매칭 | 요청에 명시된 유형 우선 |
| `doc_year` | 범위 | 최근 3년 기본, 하드 필터 없으면 전체 |
| `client_scale` | 선택적 | 요청에 명시된 경우만 적용 |

**산업군 유사성 맵** (하드 필터 완화 로직)

```python
INDUSTRY_SIMILARITY_MAP = {
    "자동차": ["모빌리티", "이동수단"],
    "가전": ["전자", "생활가전", "IT기기"],
    "식음료": ["F&B", "식품", "음료"],
    "패션": ["뷰티", "라이프스타일"],
    "금융": ["보험", "핀테크"],
    "게임": ["엔터테인먼트", "미디어"]
}
```

### 4.3 임베딩 모델 선택

**선택안 비교**

| 모델 | 다국어 | 차원 | 비용 | 추천 이유 |
|------|--------|------|------|----------|
| `text-embedding-3-large` (OpenAI) | ✅ | 3072 | $0.13/1M tokens | 높은 한국어 품질, 업계 표준 |
| `text-embedding-3-small` (OpenAI) | ✅ | 1536 | $0.02/1M tokens | 비용 효율적, 품질 충분 |
| `multilingual-e5-large` (HuggingFace) | ✅ | 1024 | 무료 (로컬) | 오프라인 가능, API 비용 없음 |
| `embed-multilingual-v3.0` (Cohere) | ✅ | 1024 | $0.10/1M tokens | 검색 특화 |

**초안 선택: `text-embedding-3-small` (OpenAI)**  
이유: 한국어 품질 검증됨, 비용 대비 성능 우수, MVP에서 충분, API 호환성 높음

> ⚠️ **확인 필요**: 로컬/오프라인 처리가 필요한 경우(내부 자료 보안 등) `multilingual-e5-large` 로 전환. 기술 스택 섹션 참조.

### 4.4 유사도 계산 및 top-k 선정 로직

```python
def hybrid_score(doc, query_metadata, semantic_sim):
    """
    metadata_score: 메타데이터 필드 매칭 점수
    semantic_score: 코사인 유사도 (0~1)
    alpha: 메타데이터 가중치 (0.4 기본값)
    """
    meta_fields = ["industry", "project_type", "proposal_purpose", "client_type"]
    match_count = sum(1 for f in meta_fields if doc[f] == query_metadata.get(f))
    metadata_score = match_count / len(meta_fields)

    alpha = 0.4
    return alpha * metadata_score + (1 - alpha) * semantic_sim

# top-k 선정: 하이브리드 점수 기준 상위 3개
# 단, 동일 문서의 복수 청크는 문서 수준으로 집계 (max pooling)
```

### 5.2 초안 생성 워크플로우

```
[Phase 1] 구조 분석
LLM 호출: 추천된 top-3 문서의 섹션 구조 분석
→ 신규 요청에 최적화된 슬라이드 구성 결정
→ 각 슬라이드에 어떤 레퍼런스 문서의 어떤 섹션을 참조할지 매핑

[Phase 2] 섹션별 콘텐츠 생성
각 슬라이드에 대해 LLM 호출:
→ 참조 섹션 텍스트 + 신규 요청 요건 → 슬라이드 텍스트 생성

[Phase 3] PPT 조립
python-pptx로 디자인 템플릿에 텍스트 삽입
→ 자리표시자(placeholder) 기반 매핑
→ output/proposal_draft_YYYYMMDD_HHMMSS.pptx 저장
```

- **구조 참조**: 레퍼런스 문서의 섹션 순서·구성 방식을 신규 제안 구조 결정에 활용

**목표**: 실제 데이터 없이도 파이프라인 테스트 가능한 환경 구축

**완료 기준**:
- [ ] Python 가상환경, 의존성 설치 완료
- [ ] Mock 제안서 20개 (텍스트 기반) + 메타데이터 JSON 생성
- [ ] ChromaDB 로컬 인스턴스 정상 기동
- [ ] OpenAI/Anthropic API 키 연결 확인

---

### Phase 1 — 인덱싱 파이프라인 (1일)

**목표**: 과거 제안서 → Vector Store 자동 인덱싱

**완료 기준**:
- [ ] Ingestor: PDF/PPTX/DOCX 파싱 및 섹션 단위 청킹
- [ ] 메타데이터 JSON 로딩 및 검증
- [ ] 임베딩 생성 및 ChromaDB 저장
- [ ] `python ingest.py --data-dir ./data/proposals` 명령어 1회 실행으로 전체 인덱싱 완료

---

### Phase 2 — 추천(검색) 모듈 MVP (1일)

**목표**: 자유 서술 입력 → top-3 레퍼런스 추천

**완료 기준**:
- [ ] Request Analyzer: 자유 서술 → 구조화 JSON 변환 (정확도 ≥ 90%, 검증 시나리오 3종)
- [ ] Hybrid Retriever: 메타데이터 필터 + 벡터 유사도 결합
- [ ] 검증 시나리오 3종에 대해 산업군·프로젝트 유형 매칭 ≥ 80%
- [ ] `retrieval_log.json` 출력 확인

---

### Phase 3 — 초안 생성 모듈 MVP (1.5일)

**목표**: top-3 레퍼런스 → PPT 초안 자동 생성

**완료 기준**:
- [ ] Draft Generator: 섹션별 텍스트 생성 (11개 슬라이드 전체)
- [ ] PPT Assembler: 디자인 템플릿에 생성 텍스트 삽입
- [ ] 검증 시나리오 3종에 대해 .pptx 파일 정상 생성
- [ ] 생성된 초안이 레퍼런스를 참조했음을 근거 로그에서 확인 가능

---

### Phase 4 — 통합 및 데모 UI (1일)

**목표**: 검증 시나리오 3종 완전 실행, 데모 시연 가능

**완료 기준**:
- [ ] CLI 단일 명령어로 전체 워크플로우 실행: `python run.py --scenario "자동차 신차 XR 체험"`
- [ ] Streamlit UI 기본 화면 (입력 폼 + 결과 뷰어 + 파일 다운로드)
- [ ] 검증 시나리오 3종 데모 영상/스크린샷 캡처
- [ ] 처리 시간 측정 (목표: 입력부터 .pptx 생성까지 ≤ 3분)

---

### Phase 5 — 고도화 (선택, 여유 시간 발생 시)

- 실제 제안서 데이터 수령 후 실데이터 인덱싱
- 디자인 양식 파일 수령 후 PPT 템플릿 정교화
- 추천 근거 설명 기능 (어떤 레퍼런스가 왜 선택됐는지 사용자에게 표시)
- 섹션별 재생성 기능 (특정 슬라이드만 다시 생성)
- 피드백 루프 (추천 결과 평가 → 향후 검색 품질 개선)

## 8. 평가 기준 대응표

### 기업부문 심사 (70%)

| 평가 항목 | 배점 | 이 설계의 대응 |
|----------|------|-------------|
| 현업 문제 이해도 | 15점 | 올림플래닛의 실제 업무 플로우(제안→수주→구현)를 분석하여, "자료 부재"가 아닌 "자산 재연결 부재" 문제를 정확히 진단. 담당자 시간 낭비 구체화 (수 시간 → 수십 분 목표). |
| 실제 도입 가능성 | 30점 | 기존 제안서 파일(PDF/PPTX/DOCX) 그대로 인입 가능. 단일 명령어 인덱싱. 결과물이 기존 작업 도구(PowerPoint)와 동일한 포맷. 추가 인프라 불필요 (로컬 실행). |
| 문제 해결의 참신성 | 40점 | 단순 키워드 검색이 아닌 메타데이터 필터링 + 시맨틱 검색의 하이브리드 구조. 레퍼런스 '참조'에 그치지 않고 신규 요청 맥락에 맞게 '재구성'하는 2단계 구조. 결과물이 바로 사용 가능한 PPT. |
| 업무 효율·비용 절감 | 15점 | 초안 작성 소요 시간 70% 단축 목표. 레퍼런스 탐색 자동화로 담당자 집중력을 아이디어·보완에 투입 가능. 반복 작업 구조화로 팀 전체 노하우 공유 효과. |

### AI전문가 심사 (30%)

| 평가 항목 | 배점 | 이 설계의 대응 |
|----------|------|-------------|
| AI 활용 적합성 | 20점 | LLM 2단계 호출(구조화 파싱 + 섹션별 생성)로 역할 분리. 임베딩 기반 시맨틱 검색으로 키워드 검색 한계 극복. 제안서 특성(구조화된 장문 한국어)에 적합한 Claude claude-sonnet-4-6 선택. |
| 기술 구현 완성도 | 20점 | 검증 시나리오 3종에 대해 end-to-end 실행 가능한 데모 제공. 오류 처리, 로깅, retrieval_log.json 근거 추적 포함. |
| 데이터·프롬프트·워크플로우 설계 | 40점 | 메타데이터 필드 11개 정의, 하이브리드 스코어링 수식 명시, 청킹 전략 근거 설명. Prompt A/B 구조 설계, 역할·규칙·출력 형식 명시. 5단계 온라인 파이프라인 단계별 입출력 정의. |
| 기술 확장 가능성 | 20점 | 신규 제안서 추가 시 `ingest.py` 재실행으로 자동 편입. 벡터 스토어를 Pinecone으로 교체 가능(인터페이스 분리). 임베딩/LLM 모델 교체 시 설정값만 변경. 웹 API 전환 시 FastAPI 래핑 구조 준비. |
