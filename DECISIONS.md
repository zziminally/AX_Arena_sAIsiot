## [2026-06-20] 하이브리드 검색 전략 채택

**결정**: 단순 벡터 유사도 검색 대신 메타데이터 필터링 + 벡터 유사도의 하이브리드 스코어링 채택  
**이유**: 제안서는 산업군·프로젝트 유형 같은 구조화된 속성이 검색 적합성에 결정적. 순수 시맨틱 검색은 "자동차 XR 제안"과 "가전 XR 제안"을 잘못 혼합할 수 있음  
**고려했던 대안**: 순수 벡터 검색만 → 메타데이터 기반 유사성을 반영 못함 / 순수 키워드 검색만 → 표현 다양성(동의어, 맥락)을 포착 못함  
**영향**: Retriever 모듈에서 두 점수(α=0.4 메타데이터, 0.6 시맨틱)를 결합하는 하이브리드 스코어링 함수 구현 필요

## [2026-06-20] 섹션 단위 청킹 채택

**결정**: 문서 전체를 단일 청크로 쓰지 않고, 섹션 단위(배경분석, 솔루션개요 등)로 청킹  
**이유**: 제안서는 명확한 섹션 구조를 가지며, 초안 생성 시 "이 섹션은 A 문서 참조, 저 섹션은 B 문서 참조"처럼 세밀한 재사용이 가능해짐  
**고려했던 대안**: 고정 토큰 수 청킹(512/1024 tokens) → 섹션 경계를 무시하여 의미 단위 훼손  
**영향**: Ingestor가 섹션 경계를 인식하는 파싱 로직 필요. Mock 데이터도 섹션 구분 포함하여 설계

## [2026-06-20] LLM 2단계 호출 구조 결정

**결정**: LLM 호출을 1차(Request Analyzer: 입력 구조화)와 2차(Draft Generator: 섹션별 생성)로 분리  
**이유**: 단일 프롬프트로 "입력 파싱 + 11개 슬라이드 생성"을 동시에 요청하면 컨텍스트 혼재로 품질 저하. 역할 분리로 각 LLM 호출의 집중도 향상  
**고려했던 대안**: 단일 거대 프롬프트 → 품질 불안정, 디버깅 어려움  
**영향**: API 호출 횟수 증가(1 + 섹션 수)이나, 섹션별 품질 제어 및 재생성 가능성 확보

## [2026-06-20] Draft Generator — 단일 LLM 호출로 전체 섹션 생성

**결정**: 섹션별 LLM 호출 대신 단일 Claude 호출로 8개 섹션 전체를 한 번에 생성  
**이유**: 섹션별 호출(8회)은 일관성 단절 위험 + 레이트 리밋 + 총 시간 증가. 단일 4096 토큰 호출이면 충분하며 문서 전체 논리 흐름이 자연스러워짐  
**고려했던 대안**: 섹션마다 별도 호출 → API 비용 8× + 섹션 간 톤 불일치 위험 / 스트리밍으로 실시간 출력 → 현 단계 불필요  
**영향**: generator.py에서 max_tokens=4096 단일 호출, 전체 JSON 파싱. JSON 절단 리스크 → max_tokens 여유 확보로 대응

## [2026-06-20] PPT 텍스트 교체 전략 — lxml 직접 조작 + python-pptx 재삽입

**결정**: 기존 텍스트 교체 시 python-pptx의 `run.text =` 대신 lxml로 `<a:p>` 전체 삭제 후 python-pptx API로 재삽입  
**이유**: 원본 PPTX에서 텍스트 양이 달라지면 기존 `<a:r>` 노드 수가 부족하거나 초과하여 레이아웃 깨짐. 노드를 완전히 제거하고 새로 만드는 것이 유일하게 안전한 방법  
**고려했던 대안**: 단순 `run.text =` 교체 → XML 노드 수 불일치로 텍스트 짤림 / 새 PPTX 전체 생성 → 디자인 손실  
**영향**: `_overwrite_frame()` 함수가 lxml + python-pptx 혼용. 첫 번째 run에서 font size/bold 캡처 후 복원

## [2026-06-20] 고도화 1단계 — 메타데이터 스코어링 개선

**결정**: 키워드 매칭을 정확 문자열 → 토큰 레벨 soft matching으로 전환 + 톤앤매너 스코어 추가 + 섹션 집계를 max pooling → top-2 평균으로 변경  
**이유**:  
- "팝업 스토어" ≠ "F&B 팝업" (정확 매칭 실패) → 가장 관련 있는 OP-S3-02가 keyword score=0을 받는 버그  
- max pooling은 섹션 하나가 우연히 높으면 문서 전체 점수를 올림 → OP-S4-02(확장 레퍼런스)가 XR 섹션 하나 덕분에 자동차·가전 top-3에 침투  
- 톤앤매너는 의뢰 맥락과 문서 스타일 적합도를 반영하는 중요 신호인데 스코어에서 빠져 있었음  
**고려했던 대안**: fuzzy string matching(Levenshtein) → 한국어 + 영문 혼용 구조에서 노이즈 많음. 토큰 split이 더 단순하고 정확  
**결과**: 자동차·가전 시나리오 3번째 슬롯이 확장 레퍼런스(OP-S4-02) → 같은 산업군 문서로 교체됨. P@3=1.0, MRR=1.0  
**영향**: 가중치 재조정 — industry(0.50), project_type(0.25), key_needs(0.15), tone(0.10). ALPHA sweep에서 결과가 alpha-insensitive하여 0.4 유지

## [2026-06-20] 고도화 2단계 — 안정성: 검증·재시도·에러 핸들링 분리

**결정**: LLM 출력 검증 로직을 `src/validator.py`로 분리하고, analyzer/generator에 재시도 로직을 추가. 파이프라인 각 단계에 try/except  
**이유**: 단일 LLM 호출에 검증이 없으면 JSON 파싱 실패 시 스택트레이스로 데모가 중단됨. 특히 발표 상황에서 치명적  
**고려했던 대안**: Pydantic 스키마 → 과도한 의존성 추가 대비 효과 미미 / LangChain output parser → 이미 Anthropic SDK 직접 사용 중이라 일관성 깨짐  
**재시도 횟수**: analyzer 최대 3회, generator 메인 최대 3회. 슬라이드2는 best-effort (실패해도 진행)  
**영향**: `validator.py` 신규 추가. 성공적인 LLM 호출에는 오버헤드 없음
